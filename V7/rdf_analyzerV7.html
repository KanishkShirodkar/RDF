RDF v4 — Complete Log Line → Code Path Mapping

Every log line emitted by rdf-flood-v4.c mapped to its exact C code location and what it means for packet fate analysis.

✅ DELIVERED path — packet reaches node fresh

// udp_rx_callback() — first receive of (src, seq) ARRIVE src=X seq=N hop=H sender_xy=(...) is_new=1 ← is_new=1 means seen[src][seq%256]==0 seen[src][seq%256] = 1; recv_count[src]++; RX src=X seq=N hop=H src_xy=(...) sender_xy=(...) my_xy=(...) ← logged only for is_new=1 PDR src=X recv=R/E = P% DISSEM src=X seq=N node=ME recv_by_me=R last_seq=L // Then pending buffer check: if(st->timer_active && st->has_pending) { if(st->pending_seq == seq) → CBF_SUPPRESS (Case A) if(seq > st->pending_seq) → PENDING_UPDATE (Case B) else → PENDING_DROP_STALE (Case C) } // No pending conflict: SUPPRESS_MISS src=X seq=N timer_active=0 has_pending=0 ... ← normal path CBF_TIMER src=X seq=N hop=H ... dist_wait_ticks=D jitter_ticks=J total_ticks=T // Timer fires → rdf_forward_callback(): FWD src=X seq=N hop=H+1 sender_xy=(...)

Count rule: Count = number of RX lines with is_new=1 at this node for this source. These are packets that arrived fresh, were accepted, and scheduled for forwarding.

🔄 PENDING UPDATED — fresher seq replaces buffered seq

// udp_rx_callback() — is_new=1, but timer already running for this src if(st->timer_active && st->has_pending) { if(seq > st->pending_seq) { // Case B (FIX-5) PENDING_UPDATE src=X old_seq=OLD new_seq=NEW timer_kept=1 st->pending_msg = msg; // replace buffered packet st->pending_seq = seq; // update seq return; // timer NOT reset — keeps running } }

What this means: Node had seq=OLD waiting in its CBF buffer. Before the timer fired, seq=NEW (fresher) arrived. The buffer is silently updated to NEW. When the timer eventually fires, it will forward NEW instead of OLD. OLD is effectively lost — it will never be forwarded. This is intentional: RDF prefers freshness over completeness.
Count rule: Count = number of PENDING_UPDATE lines at this node for this source.

🗑 STALE DROP — arrived seq is older than buffered seq

// udp_rx_callback() — is_new=1, timer running, arrived seq < pending_seq if(st->timer_active && st->has_pending) { if(seq > st->pending_seq) → PENDING_UPDATE // else: seq < pending_seq → Case C (FIX-5) PENDING_DROP_STALE src=X buffered_seq=B stale_seq=S return; // drop silently, keep existing pending }

What this means: Node already has a newer packet buffered (buffered_seq > stale_seq). The arriving packet is older — it is dropped immediately without touching the timer or buffer. This is NOT the same as CBF_SUPPRESS: suppress cancels a pending forward; stale drop just discards the incoming packet while keeping the existing pending forward intact.
Count rule: Count = number of PENDING_DROP_STALE lines at this node for this source.

🚫 CBF SUPPRESSED — duplicate cancels pending forward

// Two paths lead to CBF_SUPPRESS: // Path 1: DUPLICATE arrival (is_new=0) → rdf_cancel_pending() if(!is_new) { rdf_cancel_pending(src, seq); // if timer active AND pending_seq==seq CBF_SUPPRESS src=X seq=N ← emitted inside rdf_cancel_pending() return; } // Path 2: FIRST receive (is_new=1), pending_seq==seq (Case A) if(st->timer_active && st->has_pending && st->pending_seq == seq) { rdf_cancel_pending(src, seq); CBF_SUPPRESS src=X seq=N return; }

What this means: The node had seq=N buffered and was about to forward it. Another copy of seq=N arrived (either a duplicate or a first-receive of same seq from different path). The CBF timer is cancelled — this node will NOT forward seq=N. This is the core CBF mechanism: nodes closer to the source suppress nodes farther away.
Count rule: Count = number of CBF_SUPPRESS lines at this node for this source.

ℹ️ SUPPRESS_MISS — normal first-receive, no pending conflict

// Logged when is_new=1 AND no active pending timer for this src // (or timer active but pending_seq != seq — handled by Cases A/B/C above) SUPPRESS_MISS src=X seq=N timer_active=0 has_pending=0 pending_seq=0 // This is NOT a problem — it just means no suppression opportunity existed // Proceeds to rdf_handle_new_packet() → CBF_TIMER → eventual FWD

What this means: Normal first-receive with no pending conflict. The name is misleading — it means "we checked for suppress opportunity and missed (none existed)". This is the happy path for a fresh packet with no competing pending forward.
